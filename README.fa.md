## 📖 درباره‌ی این مخزن

این مخزن **یک فریم‌ورک نیست**، و **یک wrapper دور API مدل زبانی هم نیست**. این یک آزمایشگاه مهندسی سیستم است که برای پاسخ دقیق به یک سؤال ساخته شده:

> **چطور می‌توان روی یک مؤلفه که ذاتاً احتمالاتی (Probabilistic) است، نرم‌افزاری قابل‌اعتماد، قابل‌ممیزی و در سطح پروداکشن ساخت؟**

هر فولدر در این مخزن، یک اثبات مفهوم (Proof of Concept) مستقل، مینیمال و قابل‌اجرا برای **یک دغدغه‌ی معماری خاص** در سیستم‌های ایجنتیک هوش مصنوعی است — کنترل وضعیت، بازیابی پس از کرش، بازتلاش (Retry)، اجرای موازی، هدایت مدل (Model Routing)، استدلال روی لبه (Edge Inference)، بازیابی دانش (Retrieval)، حافظه، ارکستریشن چند-ایجنتی، امنیت و پایش‌پذیری (Observability). کدها عمداً کوچک نگه داشته شده‌اند (۳۰ تا ۱۰۰ خط برای هر مفهوم). هدف، ساختن **مدل‌های ذهنی قابل‌انتقال (Transferable Mental Models)** است، نه تسلط بر یک فریم‌ورک خاص — بعد از این دوره باید بتوانی به هر فریم‌ورک ایجنتیکی (LangGraph، Temporal، AutoGen، CrewAI، Restate و...) نگاه کنی و فوراً تشخیص دهی کدام‌یک از این پرایمیتیوها را (دوباره) پیاده‌سازی کرده و چه Trade-offای را انتخاب کرده است.

این یک پروژه‌ی شخصی آموزش مهندسی است، که به شکل یک دوره‌ی پیش‌رونده (Progressive Curriculum) با کد مرجع قابل‌اجرا در هر توقف‌گاه ساختار یافته است.

## 🎯 فلسفه بنیادین

کل این آزمایشگاه بر پایه‌ی یک اصل موضوعه (Axiom) بنا شده که در سراسر مسیر با عنوان **AI Systems Thinking** از آن یاد می‌شود:

> **مدل‌های زبانی احتمالاتی هستند. نرم‌افزار باید قطعی باشد.**
> وظیفه‌ی تو به‌عنوان معمار سیستم این نیست که مدل را قابل‌پیش‌بینی‌تر کنی — وظیفه‌ات ساختن یک Runtime قطعی *دور* یک مؤلفه‌ی ذاتاً غیرقابل‌پیش‌بینی است، طوری که این غیرقابل‌پیش‌بینی بودن مهار، قابل‌مشاهده و قابل‌بازیابی باشد.

اگر بدون کنترل رها شود، یک ایجنت مبتنی بر LLM:
- مراحل را دور می‌زند و پیشرفتی را که واقعاً رخ نداده هالوسینیت می‌کند
- در یک حلقه‌ی بی‌نهایتِ فراخوانی ابزار (Tool-Calling Loop) گیر می‌کند
- به محض کرش یا قطع شبکه، تمام وضعیت و کانتکست را از دست می‌دهد
- در حالی که استدلالش کاملاً منحرف شده، به‌طور خاموش `200 OK` برمی‌گرداند

هر فاز در این مخزن برای بستن یکی از همین حالت‌های شکست (Failure Modes) ساخته شده است.

## 🏗️ معماری سیستم

اصلی که در تمام لایه‌های این آزمایشگاه تکرار می‌شود:

> **LLM هرگز مستقیماً با دنیای بیرون تماس ندارد.** او فقط یک نیت (Intent) را — به‌صورت یک پیشنهاد ساختاریافته و اعتبارسنجی‌شده — *پیشنهاد* می‌دهد. تنها یک Runtime قطعی، که توسط یک FSM، یک قرارداد ابزار (Tool Contract) و یک لایه‌ی پایداری (Durability Layer) محافظت می‌شود، اجازه‌ی اجرای واقعی هر کاری را دارد.

```
       ┌────────────────────────────────────────────────────────┐
       │                   DETERMINISTIC RUNTIME                │
       │                                                        │
       │   ┌──────────────┐     FSM Guard Check     ┌───────┐   │
       │   │              │ ──────────────────────► │       │   │
       │   │ Context &    │                         │ LLM   │   │
       │   │ State Store  │ ◄────────────────────── │ Engine│   │
       │   └──────────────┘    Structured JSON      └───────┘   │
       │          ▲                                     │       │
       │          │          Durable Activity           │       │
       │          └─────────────────────────────────────┘       │
       │                        Tool Result                     │
       └────────────────────────────────────────────────────────┘
```

این «ساندویچ Runtime» — **کنترل FSM ← نیت اعتبارسنجی‌شده ← اجرای پایدار و هم‌پوتانس ← Checkpoint ← پایش** — الگویی است که هر فاز بعدی (Routing، RAG، حافظه، Multi-Agent، امنیت) روی آن سوار می‌شود.

<summary><h3>فاز ۲ — FSM & State</h3></summary>

**📁 `fsm/`**

یک ماشین وضعیت متناهی (Finite State Machine) مثل یک فرمانده‌ی سخت‌گیر بالای سر LLM عمل می‌کند: سیستم در هر لحظه دقیقاً در یک وضعیت قرار دارد، و فقط زمانی می‌تواند به وضعیت بعدی برود که یک رویداد *مجاز* رخ داده باشد.

| عنصر | تعریف | مثال |
|---|---|---|
| **State (وضعیت)** | حقیقتِ کنونیِ سیستم | `READING` |
| **Event (رویداد)** | سیگنالی که نشان می‌دهد اتفاقی افتاده | `FILE_FOUND` |
| **Transition (انتقال)** | قانون جابه‌جایی بین وضعیت‌ها | `READING --FILE_FOUND--> ANALYZING` |
| **Invariant (خط قرمز)** | قانونی که تحت هیچ شرایطی نباید بشکند | «رفتن به REPORT قبل از خواندن فایل ممنوع است» |

دو لایه‌ی کنترلی این FSM را از یک جدول ساده فراتر می‌برند:
- **Guard (شرط انتقال)** — یک شرط بولی که *پیش* از اجرای انتقال بررسی می‌شود (مثلاً رد کردن `ANALYZING` اگر فایل خواندن‌شده ۰ بایت بوده باشد)
- **Invariant (اصل دست‌نخورده)** — قانونی که در کل چرخه‌ی حیات سیستم اجرا می‌شود (مثلاً «مجموع Retryها در تمام حالت‌ها ≤ ۳»)

یک رویداد خارج از نوبت (مثلاً رسیدن `REPORT_GENERATED` در حالی که سیستم در `READING` است) با صدور یک Exception **رد** می‌شود — FSM هرگز اجازه‌ی میانبر زدن یا ورود داده‌ی نامعتبر را نمی‌دهد.

**Persistence مستقیماً به FSM متصل شده است**: هر انتقال، *به‌صورت سنکرون و پیش از* اجرای مرحله‌ی بعدی، روی دیسک Checkpoint می‌شود. با روشن شدن مجدد سرور، FSM آخرین وضعیت شناخته‌شده را از دیسک می‌خواند و دقیقاً از همان‌جا ادامه می‌دهد — بدون دوباره‌کاری، بدون از دست رفتن کانتکست.

```python
def transition_to(self, new_state: State, guard: bool = True) -> None:
    if new_state not in self.TRANSITIONS[self.state]:
        raise ValueError(f"Illegal transition: {self.state} -> {new_state}")
    if not guard:
        raise PermissionError(f"Guard failed for target state: {new_state}")
    self.state = new_state
    self._checkpoint()  # دیسک قبل از اینکه به وضعیت درون‌حافظه اعتماد شود، به‌روز می‌شود
```

**مدیریت خطا در سطح FSM:** در صورت بروز خطا، سیستم به یکی از سه مسیر هدایت می‌شود: `Retry` (گذرا/Transient)، `Fail` (قطعی/دائمی)، یا `Escalate` (ارجاع به انسان) — این دسته‌بندی به‌طور کامل در فاز ۴ باز می‌شود.

<summary><h3>فاز ۳ — Durable Execution</h3></summary>

**📁 `durable_execution/`**

مسئله: یک پایپ‌لاین چندمرحله‌ای (استخراج ← خلاصه‌سازی توسط LLM ← ذخیره در دیتابیس) به‌دلیل یک قطعی کوتاه شبکه، وسط کار کرش می‌کند. با اجرای مجدد ساده، LLM *دوباره* فراخوانی می‌شود — که هم توکن و زمان تلف می‌کند، و هم (چون LLMها غیرقطعی‌اند) ممکن است بار دوم پاسخ **متفاوتی** بگیریم.

**Durable Execution** (الگویی که Temporal و Restate بر پایه‌ی آن ساخته شده‌اند) به تابع Workflow شما حافظه‌ای می‌دهد که در برابر کرش مقاوم است:

> **قانون طلایی:** در بازپخش (Replay)، مراحلی که قبلاً با موفقیت اجرا شده‌اند هرگز دوباره اجرا نمی‌شوند — نتیجه‌ی قبلی‌شان از دیسک خوانده و تزریق می‌شود (Memoization / Replay).

معماری به دو نقش کاملاً مجزا تقسیم می‌شود:

- **Workflow (ارکستریتور)** — باید **۱۰۰٪ قطعی (Deterministic)** باشد. نه `time.now()`، نه `random()`، نه فراخوانی مستقیم HTTP. فقط منطق ترتیب اجرا.
- **Activity (عامل اجرایی)** — تنها بخشی که اجازه‌ی تماس با دنیای بیرون را دارد (فراخوانی LLM، I/O دیسک، نوشتن در دیتابیس). هر Activity به‌طور مستقل Retry و Cache می‌شود.

```python
def run_activity(step_id: str, func, *args):
    log = json.loads(LOG_FILE.read_text()) if LOG_FILE.exists() else {}
    if step_id in log:
        return log[step_id]              # REPLAY — اجرای واقعی Skip می‌شود
    result = func(*args)                 # EXECUTE — اجرای واقعی، اولین بار
    log[step_id] = result
    LOG_FILE.write_text(json.dumps(log, indent=2))
    return result
```

**چرا جداسازی Workflow/Activity قابل‌مذاکره نیست:** اگر `datetime.now()` یا `random()` را مستقیماً داخل بدنه‌ی Workflow قرار دهید، مسیر تصمیم‌گیری در هر Replay تغییر می‌کند، و موتور ارکستریشن دیگر نمی‌تواند متغیرهای در حال اجرا را با رویدادهایی که قبلاً روی دیسک لاگ کرده تطبیق دهد. این یک **باگ خاموش و سخت‌کشف در Replay غیرقطعی** است — یکی از رایج‌ترین اشتباهات هنگام ساخت این الگو از صفر.

<details>
<summary><h3>فاز ۴ — Failure, Retry & Idempotency</h3></summary>

**📁 `Failure-Retry-Idempotency/`**

همه‌ی خطاها یکسان نیستند. یکسان فرض کردن آن‌ها اولین اشتباهی است که اکثر مهندسان مرتکب می‌شوند:

| نوع خطا | ماهیت | نمونه | استراتژی |
|---|---|---|---|
| **Transient (گذرا)** | موقتی و غیرقطعی | `503`، `429 Rate Limit` | Retry با Backoff |
| **Permanent (ثابت)** | قطعی، هرگز موفق نمی‌شود | `401 Auth Failure`، خطای Validation | شکست فوری — بدون Retry |
| **Poison Pill (سمی)** | خودِ ورودی باعث کرش مکرر می‌شود | JSON خراب، Overflow | Quarantine / ارجاع |

**Exponential Backoff + Jitter** از پدیده‌ی *Thundering Herd* جلوگیری می‌کند — اگر ۱۰ ایجنت هم‌زمان به خطای `429` بخورند و همه دقیقاً بعد از ۱ ثانیه Retry کنند، به‌طور هم‌زمان دوباره سرور مقصد را زیر بار می‌برند. Backoff زمان انتظار را به‌صورت نمایی بالا می‌برد (`1s → 2s → 4s → 8s`)؛ Jitter با اغتشاش تصادفی، ایجنت‌ها را بیشتر از هم فاصله می‌دهد.

**Idempotency** نیمه‌ی دیگر Retry ایمن است:

> عملیاتی Idempotent است اگر اجرای N باره‌ی آن، دقیقاً همان اثری را بگذارد که اجرای ۱ باره‌اش می‌گذارد.

بدون این اصل، Retry یک درخواست «ارسال ایمیل» یا «پردازش پرداخت» می‌تواند یک Side Effect که قبلاً موفق شده را دوباره اجرا کند — درخواست انجام شده، اما پاسخ در قطعی شبکه گم شده و باعث یک Retry بی‌مورد شده است. راهکار، یک **Idempotency Key** (مثلاً `hash(task_id + step_id)`) است که به هر درخواست ضمیمه می‌شود؛ اگر لایه‌ی گیرنده قبلاً این کلید را پردازش کرده باشد، به‌جای اجرای مجدد، همان نتیجه‌ی کش‌شده را برمی‌گرداند.

```python
sleep_time = random.uniform(0, min(8.0, base_delay * (2 ** attempt)))  # Full Jitter
```

**بستن زیربنای قطعی:** فازهای ۱ تا ۴ با هم یک *Runtime مقاوم در برابر کرش* می‌سازند — از این پس هر LLMای می‌تواند روی این شاسی سوار شود، بدون ترس از قطع شبکه، اتلاف مالی ناشی از تکرار Side Effectها، یا رفتار آنارشیستیک ناشی از دور زدن مراحل.

<summary><h3>فاز ۵ — DAG & Workflow Execution</h3></summary>

**📁 `dag/`**

اگر FSM «قطب‌نما و ناظر چرخه‌ی حیات» باشد، DAG «موتور نقشه‌راه و اجرای موازی» است. یک وضعیت واحد FSM (مثلاً `EXECUTE`) غالباً چند زیروظیفه‌ی مستقل را پنهان می‌کند — واکشی ۵ سند، ترکیب نتایج، اجرای کد اعتبارسنجی. اجرای خطی این مراحل، Latency را نابود می‌کند؛ و FSM به‌تنهایی هیچ مفهومی از «این کارها را هم‌زمان اجرا کن» ندارد.

یک **گراف جهت‌دار بدون چرخه (DAG)** از این دو عنصر تشکیل می‌شود:
- **Nodes (گره‌ها)** — Activityهای اجرایی (فراخوانی API، Query دیتابیس، پردازش متن)
- **Edges (یال‌ها)** — وابستگی‌های داده‌ای یا ترتیبی بین گره‌ها

اگر `Task_A` و `Task_B` هیچ وابستگی به هم نداشته باشند، کاملاً **موازی** اجرا می‌شوند. `Task_C` که به هر دو وابسته است، منتظر یک **Fan-in Join** می‌ماند. «Acyclic» یعنی هیچ مسیری نباید به گره‌ی شروع برگردد — این باعث Deadlock می‌شود.

> **اصل معماری:** هر گره یک **تابع خالص (Pure Function)** است — ورودی را فقط‌خواندنی می‌گیرد، خروجی جدید برمی‌گرداند. این موتور DAG است (نه یک حافظه‌ی مشترک قابل‌تغییر) که مسئول تزریق خروجی هر گره به ورودی گره‌ی بعدی است. دستکاری یک دیکشنری مشترک بین شاخه‌های موازی، رایج‌ترین راه بروز Race Condition در این‌جاست.

**حالت‌های شکست خاص DAG:**
- **ایزوله‌سازی شاخه‌ها (Branch Isolation)** — اگر یکی از ۳ شاخه‌ی موازی شکست بخورد، آیا کل DAG را `Short-circuit` (لغو) می‌کنید (مناسب کارهای کاملاً وابسته) یا `Best-effort` پیش می‌روید (مناسب کاری مثل خلاصه‌سازی LLM که با ۸۰٪ داده هم کار می‌کند)؟
- **DAG استاتیک در برابر پویا** — یک گراف استاتیک پیش از اجرا کاملاً مشخص است (`Fetch → Process → Save`). یک **DAG پویا/Agentic**، که خود LLM حین اجرا تصمیم می‌گیرد چه گره‌هایی اضافه کند، مدیریتش بسیار سخت‌تر است چون LLM ممکن است ناخواسته یک چرخه ایجاد کند — و نیازمند یک Guard است که چرخه‌ها را حین اجرا شناسایی و رد کند.

```python
ready_nodes = [
    node for name, node in self.nodes.items()
    if name not in completed and node.dependencies.issubset(completed)
]
tasks = [self._run_node(node, results) for node in ready_nodes]
node_results = await asyncio.gather(*tasks)   # Fan-out واقعی
```

<summary><h3>فاز ۶ — Agent Runtime</h3></summary>

**📁 `agent-runtime/`**

با استقرار تمام لایه‌های قطعی (FSM، Durable Execution، Retry/Idempotency، DAG)، فاز ۶ اولین‌باری است که LLM به‌عنوان **تصمیم‌گیرنده** روی این شاسی سوار می‌شود.

یک Agent Runtime، حلقه‌ی کنترلی است که الگوی **مشاهده ← تفکر ← اقدام ← راستی‌آزمایی** (الگوی ReAct) را پیاده می‌کند، و از چهار ماژول تشکیل شده است:

- **Context Manager (مدیر زمینه)** — پرومپت سیستم، تاریخچه‌ی فراخوانی ابزارها و وضعیت فعلی FSM را برای LLM گردهم می‌آورد
- **Intent Parser & Validator (اعتبارسنج نیت)** — خروجی احتمالاتی LLM را می‌گیرد و از فیلتر یک Pydantic Schema عبور می‌دهد
- **FSM Guardrail (سپر FSM)** — بررسی می‌کند آیا اکشن درخواستی LLM اصلاً در وضعیت فعلی FSM مجاز است یا نه
- **Tool Dispatcher (توزیع‌کننده ابزار)** — اکشن تأییدشده را به‌عنوان یک Activity پایدار (قابل Retry، Idempotent، قابل Cache) اجرا می‌کند

> **اصل غیرقابل‌مذاکره:** LLM هرگز مستقیماً یک ابزار را صدا نمی‌زند یا دیتابیسی را تغییر نمی‌دهد. او فقط یک نیت را *پیشنهاد* می‌دهد. این Runtime است که آن پیشنهاد را ارزیابی، ثبت و سپس درون یک بستر پایدار اجرا می‌کند.

حالت‌های شکست حیاتی که این فاز در برابرشان محافظت می‌کند شامل **Death Spiral** (ایجنتی که بی‌نهایت یک ابزار را دوباره صدا می‌زند) و **Unbounded Autonomy** هستند — هر دو با ترکیب بررسی‌های FSM Guard و یک سقف سخت که توسط Tool Dispatcher اعمال می‌شود، مهار می‌شوند.

<summary><h3>فاز ۷ — Observability & Evals (نسخه‌ی اول)</h3></summary>

**📁 `Observability-Eval-Engine/`**

مانیتورینگ سنتی APM (مثل Datadog، Prometheus که حول کد وضعیت HTTP، CPU و Latency می‌چرخد) برای سیستم‌های ایجنتیک شکست می‌خورد، چون شکست‌ها اینجا **خاموش (Silent)** هستند: یک API می‌تواند `200 OK` برگرداند، هیچ Exceptionای صادر نکند، و در عین حال پاسخ کاملاً هالوسینیت‌شده تولید کرده یا ۳ برابر توکن مورد انتظار مصرف کرده باشد.

> **اصل:** در سیستم‌های AI، ما کد را لاگ نمی‌کنیم — بلکه *مسیر استدلال* (Graph of Thought) و *تغییرات State* را Trace می‌کنیم.

دو مفهوم مکمل:

| مفهوم | چیست |
|---|---|
| **Observability (Tracing)** | پایش زنده در Production — یک `Trace` کل یک اجرا است؛ یک `Span` هر زیرعملیات (یک فراخوانی LLM، یک Tool Call) |
| **Evaluations (Evals)** | جایگزین Unit Test برای رفتار غیرقطعی — آیا خروجی صحیح، وفادار به داده، و مطابق پرومپت مانده است؟ |

یک **موتور ارزیابی دو لایه‌ای**:
- **Deterministic Evals** (سریع، ارزان) — اعتبارسنجی Schema، آستانه‌ی Latency، بررسی کلمات ممنوعه
- **Probabilistic Evals / LLM-as-a-Judge** — یک مدل مجزا امتیاز **Faithfulness** (آیا پاسخ در داده‌های ورودی ریشه دارد یا ساختگی است؟) و **Answer Relevance** را محاسبه می‌کند

**ریسک‌های عملیاتی که باید حول آن‌ها طراحی کرد:**
- **نشت PII در Trace‌ها** — ارسال داده‌های خام کاربر/API Key به سیستم لاگینگ خارجی بدون یک لایه‌ی Redaction
- **قانون Goodhart در LLM-as-a-Judge** — بهینه‌سازی صرف برای بالا بردن امتیاز داور، پاسخ‌های قشنگ اما بی‌محتوا تولید می‌کند
- **سربار Telemetry** — لاگ کردن پرومپت‌های چندکیلوبایتی در هر فراخوانی، خودش Latency و هزینه‌ی ذخیره‌سازی واقعی اضافه می‌کند

<summary><h3>فاز ۸ — Model Routing</h3></summary>

**📁 `model-routing/`**

ارسال تمام درخواست‌ها — از یک دسته‌بندی ساده تا استدلال ۵ مرحله‌ای پیچیده — به یک مدل سنگین (GPT-4o، Claude Sonnet) یک آنتی‌پترن پرهزینه و Latency-زا است. یک **Model Router** یک لایه‌ی تصمیم‌گیری ارزان و فوق‌سریع (معمولاً زیر ۲۰ میلی‌ثانیه) است که پیش از فراخوانی اصلی LLM قرار می‌گیرد و بر اساس پیچیدگی، نیاز به Schema، SLA زمانی و بودجه‌ی هزینه، مسیر را تعیین می‌کند.

```
                      MODEL ROUTER
 User Request ──► 1. Heuristic Check (Regex/Length)
                  2. Semantic Embedding Distance
                  3. SLA & Cost Policy Evaluation
                       │
     ┌─────────────────┼─────────────────┐
     ▼ (ساده)            ▼ (متوسط)          ▼ (استدلال سنگین)
 Local SLM         Fast Cloud            Heavy LLM
 (Llama 8B)        (Flash / Haiku)       (Sonnet / GPT-4o)
```

سه استراتژی هدایت، از ارزان‌ترین به پیچیده‌ترین:
1. **Static / Rule-Based** — طول پرومپت، کلیدواژه‌ها، نوع ابزار درخواستی ← Latency نزدیک صفر
2. **Semantic Routing** — تبدیل پرومپت به Embedding، مقایسه‌ی فاصله‌ی کسینوسی با خوشه‌های نیت شناخته‌شده ← حدود ۱۰ تا ۳۰ میلی‌ثانیه
3. **Model-Based Cascading** — یک مدل بسیار کوچک پیچیدگی را بین `0.0` تا `1.0` امتیاز می‌دهد؛ زیر `0.3` ← لایه‌ی ارزان، بالای `0.7` ← لایه‌ی سنگین

**Fallback Cascading** به‌اندازه‌ی تصمیم اولیه‌ی Routing اهمیت دارد: با دریافت `429`، `503` یا نقض فرمت Schema از مدل اول، روتر به‌طور شفاف بدون متوقف کردن FSM، درخواست را به لایه‌ی جایگزین هدایت می‌کند.

**ریسک‌های خاص Routing:**
- **جریمه‌ی زمانی روتر (Router Latency Tax)** — اگر خود روتر ۲۰۰ میلی‌ثانیه زمان صرف کند تا شما را از یک فراخوانی ۱ ثانیه‌ای نجات دهد، ارزش افزوده از بین می‌رود
- **شکنندگی Schema در مدل‌های کوچک** — مدل‌های کوچک در تولید JSON پیچیده به‌طور معناداری ضعیف‌ترند؛ سپردن کارهای Structured Output پیچیده به آن‌ها، نرخ خطای Validation را بالا می‌برد
- **نوسان در Routing (Flapping)** — تغییرات جزئی در پرومپت کاربر نباید باعث انتخاب‌های ناسازگار سطح مدل شود، که برای کاربر به‌صورت کیفیت ناهماهنگ ظاهر می‌شود

<summary><h3>فاز ۹ — Inference & Edge AI</h3></summary>

**📁 `Inference-KV-Cache/`**

گلوگاه اصلی سخت‌افزاری: تولید توکن **محدود به پهنای باند حافظه است، نه محدود به قدرت پردازشی**. برای هر توکن، باید تمام وزن‌های مدل از RAM به داخل Cache/VRAM جابه‌جا شوند.

> **نتیجه‌ی عینی:** یک مدل 8B با دقت FP16 (حجم ۱۶ گیگابایت) روی سخت‌افزاری با پهنای باند حافظه‌ی ۴۰ گیگابایت بر ثانیه، سقف نظری `40 / 16 = 2.5 توکن در ثانیه` دارد — صرف‌نظر از تعداد هسته‌های پردازنده.

**Quantization** دقت را با فضای پهنای باند معامله می‌کند:

| دقت | حجم مدل 8B | افت Perplexity | مناسب برای |
|---|---|---|---|
| FP16 | ~۱۶ گیگابایت | ۰٪ (پایه) | سرورهای ابری |
| Q8_0 | ~۸.۵ گیگابایت | ناچیز (کمتر از ۱٪) | دقت بالای محلی |
| **Q4_K_M** | **~۴.۵ گیگابایت** | **جزئی (۱ تا ۳٪)** | **نقطه‌ی طلایی برای Edge** |
| Q2_K | ~۲.۸ گیگابایت | شدید (بیش از ۱۵٪) | غیرقابل‌استفاده برای استدلال |

`Q4_K_M` انتخاب پیش‌فرض برای استقرار روی Edge است: کاهش ~۷۵٪ حجم، با حداقل فروپاشی منطق استدلال.

**KV Cache** خروجی‌های Key/Value توجه (Attention) را ذخیره می‌کند تا مدل مجبور نباشد در هر گام کل ماتریس Attention را دوباره محاسبه کند — اما همین Cache به‌صورت خطی با طول کانتکست رشد می‌کند (یک مدل 8B با کانتکست 8k، در دقت FP16، می‌تواند به‌تنهایی ۱.۵ تا ۲ گیگابایت مصرف کند). راهکارها: KV Cache کوانتیزه‌شده (INT8/INT4) و مدیریت حافظه‌ی شبیه به PagedAttention.

**حالت‌های شکست مختص محیط Edge/موبایل (مثلاً Termux روی اندروید):**
- **OOM Killer** — اندروید در صورت اتمام RAM، فرایند را به‌طور خاموش `SIGKILL` می‌کند، بدون هیچ خطای صریح
- **Thermal Throttling (افت فرکانس حرارتی)** — اجرای طولانی روی CPU می‌تواند فرکانس هسته را تا ۵۰٪ کاهش دهد، که Latency را به‌طور خطی بالا می‌برد
- **زوال کانتکست در مدل‌های کم‌بیت** — مدل‌های کوانتیزه‌ی ۴ بیتی، در طول کانتکست‌های بالای ~۴k توکن، سریع‌تر از مدل‌های اصلی دچار حلقه‌ی تکرار کلمات می‌شوند

پیاده‌سازی مرجع اینجا یک **محاسبه‌گر بودجه‌ی حافظه** است که حجم وزن‌ها + حجم KV Cache + سربار Runtime را در برابر RAM موجود می‌سنجد و *پیش از* تلاش برای بارگذاری مدل، یک حکم قطعی `feasible: true/false` برمی‌گرداند — و یک کرش OOM در زمان اجرا را به یک بررسی در زمان طراحی تبدیل می‌کند.

<summary><h3>فاز ۱۰ — RAG & Context Management</h3></summary>

**📁 `RAG-Context-Management/`**

RAG ساده (Chunking با طول ثابت + یک Vector Store واحد) در پروداکشن به‌طور مکرر به سه دلیل شکست می‌خورد: **دقت پایین** (نویز بازیابی‌شده)، **فراخوانی ناقص** (از دست دادن کلیدواژه‌ها/شناسه‌های دقیق)، و **«گم شدن در وسط»** (LLMها به وسط یک کانتکست طولانی توجه کمتری می‌دهند).

> **اصل معماری:** لایه‌ی بازیابی دانش، یک Vector Store ساده نیست — بلکه یک پایپ‌لاین سه‌مرحله‌ای است: **Chunking ← بازیابی ترکیبی ← Re-ranking**.

```
Raw Docs → [Chunking معنایی/آگاه به AST] → [Hybrid Search: Dense + Sparse]
         → [Reciprocal Rank Fusion] → [فیلتر Cross-Encoder] → کانتکست خالص به LLM
```

- **Chunking** — آگاه به AST/کد (هرگز یک تابع را از وسط بدنه‌اش پاره نکند) یا معنایی (مرز Chunk جایی تعیین می‌شود که فاصله‌ی Embedding بین جملات متوالی از یک آستانه بگذرد)
- **Hybrid Search** — بازیابی Dense (Embedding برداری) در تشابه انتزاعی عالی است اما در شناسه‌های دقیق ضعیف؛ بازیابی Sparse (BM25/TF-IDF) دقیقاً روی کلیدواژه‌ها منطبق می‌شود اما هیچ درک معنایی ندارد. ترکیب آن‌ها از طریق **Reciprocal Rank Fusion (RRF)**:  
  `RRF_score(d) = Σ 1 / (k + rank_m(d))`، معمولاً با `k = 60`.
- **Re-ranking** — یک **Cross-Encoder** کندتر اما بسیار دقیق‌تر (مثل BGE-Reranker)، کاندیداهای Top-K را به ۵ نتیجه‌ی بسیار مرتبط فیلتر می‌کند و نویز کانتکست را تا ~۷۰٪ کاهش می‌دهد.

**حالت‌های شکست:**
- **شکستن مرز Chunk** — یک تعریف کلیدی که بین دو Chunk تقسیم شود، معنایش را از دست می‌دهد؛ راهکار یک **Parent-Document Retriever** است (جستجو روی Chunkهای کوچک، اما تزریق Chunk بزرگ‌تر/مادر به کانتکست)
- **آلودگی کانتکست** — Chunkهای نامرتبط، مدل را به‌سمت پاسخ‌های هالوسینیت‌شده سوق می‌دهند
- **انفجار Latency بازیابی** — جستجوی برداری + BM25 + Re-ranking باید Async/موازی اجرا شوند وگرنه Latency روی هم انباشته می‌شود

</details>

<summary><h3>فاز ۱۱ — Memory Systems</h3></summary>

**📁 `Memory-Systems/`**

نگه‌داشتن کل تاریخچه‌ی مکالمه در Context Window مقیاس‌پذیر نیست — محدودیت توکن، Latency و هزینه همه فرو می‌ریزند. معماری حافظه تقریباً بر روی الگوی شناختی مغز انسان نگاشته می‌شود و به چهار لایه تقسیم می‌شود:

```
                 AGENT MEMORY SYSTEM
       ┌───────────────┼───────────────┐
       ▼                ▼                ▼
 Short-Term        Semantic LTM      Episodic LTM
 (Working)          (Fact/KV)        (Experience)
 Sliding Buffer    User Profiles    Execution Logs
                   & Domain Facts   & Self-Correction
```

۱. **حافظه‌ی کوتاه‌مدت/کاری (Working/STM)** — کانتکست زنده‌ی Session جاری، مدیریت‌شده با Sliding Window و خلاصه‌سازی پویا

۲. **حافظه‌ی بلندمدت معنایی (Semantic LTM)** — حقایق و ترجیحات پایدار («زبان مورد علاقه‌ی کاربر پایتون است»)، ذخیره‌شده به‌صورت KV/گراف دانش/Document Store

۳. **حافظه‌ی بلندمدت رویدادی (Episodic LTM)** — تاریخچه‌ای از اقدامات و نتایج گذشته («در ۲ فوریه، اتصال روی پورت ۵۴۳۲ Timeout شد؛ پورت ۵۴۳۳ آن را حل کرد»)، بازیابی بر اساس تشابه + زمان

۴. **حافظه‌ی رویه‌ای (Procedural Memory)** — دستورالعمل‌های استاتیک ایجنت، System Promptها و ساختار FSM

**تثبیت حافظه (Memory Consolidation)** — فرآیند ارتقای STM به LTM — به‌صورت یک **Worker پس‌زمینه‌ی Async** اجرا می‌شود، جدا از حلقه‌ی پاسخ‌دهی به کاربر تا هیچ Latencyای به مسیر تعاملی تحمیل نشود. همه‌ی داده‌ها ارزش ورود به LTM را ندارند: هر حقیقتِ کاندید بر اساس **تازگی (Recency)**، **اهمیت (Importance، سنجیده‌شده توسط یک مدل کوچک)** و **فراوانی فراخوانی (Frequency)** امتیازدهی می‌شود.

**مدیریت تناقض** — اگر کاربر قبلاً گفته باشد «من در تهران هستم» و بعداً بگوید «به برلین منتقل شدم»، سیستم به **Temporal Versioning** (`created_at` / `valid_until` برای هر Fact) و منطق **Upsert بر پایه‌ی استخراج Entity** نیاز دارد: پیش از نوشتن، پایگاه داده بر اساس Entity استعلام می‌شود و مقدار قبلی Overwrite یا Deprecated می‌شود.

**حالت‌های شکست:**
- **مسمومیت حافظه (Memory Poisoning)** — یک حقیقت هالوسینیت‌شده که وارد LTM شود، در هر نوبت بعدی تقویت و تکرار می‌شود
- **سربار کانتکست کهنه** — تزریق بیش از حد حافظه‌های قدیمی باعث حواس‌پرتی کانتکست و افت کیفیت استدلال می‌شود
- **Race Condition ناهمگام** — دو پیام سریع کاربر می‌توانند با Worker تثبیت‌کننده مسابقه دهند و وضعیت حافظه را خراب کنند

<summary><h3>فاز ۱۲ — Multi-Agent Orchestration & Swarms</h3></summary>

**📁 `Multi-Agent-Router-Handoff-Engine/`**

یک ایجنت تک‌نفره که هم‌زمان برنامه‌ریزی، کدنویسی، اجرا، تست و اصلاح را انجام دهد، دچار **Prompt Entropy** می‌شود — با افزایش تعداد ابزارها و دستورالعمل‌ها، نرخ هالوسینیشن در انتخاب ابزار به‌طور نمایی بالا می‌رود، و تاریخچه‌ی اجرای نامرتبط (مثلاً خروجی ۱۰۰۰ خطی یک دیباگر) فضای استدلال لازم برای برنامه‌ریزی سطح بالا را آلوده می‌کند.

> **راهکار معماری:** تجزیه به شبکه‌ای از ایجنت‌های تک‌مسئولیتی با ایزوله‌سازی کانتکست و ابزارهای محدود.

دو توپولوژی غالب:

```
[سلسله‌مراتبی / Supervisor]              [Peer-to-Peer / Handoff Swarm]
        Supervisor                              Agent A
       ┌────┴────┐                                 │ (handoff)
   Worker A   Worker B                          Agent B
```

- **سلسله‌مراتبی / Supervisor** — یک ایجنت سطح‌بالا مسئله را تجزیه می‌کند، به Workerها ارجاع می‌دهد، نتایج را ترکیب می‌کند. به‌شدت قابل‌پیش‌بینی، قابل‌تست، و به‌شکل DAG. مناسب برای: پایپ‌لاین‌های خطی تولید کد، ETL، پردازش اسناد.
- **Peer-to-Peer Swarm / Handoff** — بدون Supervisor؛ هر ایجنت پس از اتمام کار، مستقیماً `transfer_to_agent(...)` را صدا می‌زند. انعطاف‌پذیری بسیار بیشتری در برابر مسیرهای خطای پیش‌بینی‌نشده دارد. مناسب برای: پشتیبانی مشتریان پیچیده، حلقه‌های خودکار تحقیق و توسعه.

**حالت‌های شکست حیاتی:**
- **حلقه‌ی Ping-Pong** — ایجنت A کار را به B پاس می‌دهد، B به‌دلیل کمبود اطلاعات آن را دوباره به A برمی‌گرداند، و این چرخه ادامه می‌یابد تا بودجه تمام شود. راهکار: یک **بودجه‌ی سختِ سراسری برای Handoff** و شناسایی چرخه با پیچیدگی O(V+E) روی گراف ارجاعات.
- **نشت State و انباشت کانتکست** — انتقال کامل تاریخچه‌ی مکالمه در هر Handoff، اصل ایزوله‌سازی را زیر سؤال می‌برد. راهکار: **قراردادهای صریح و کوچک Payload بر پایه‌ی Pydantic** بین ایجنت‌ها، نه انتقال کامل تاریخچه.
- **شکست زنجیره‌ای (Cascade Failure)** — خطا در خروجی ایجنت اول (مثلاً Planner) به‌عنوان فرض درست وارد ایجنت دوم می‌شود و کل مسیر را منحرف می‌کند.

```python
if current_agent_name in handoff_history[-2:]:
    raise RuntimeError(f"[SWARM CRASH] Ping-pong loop detected on '{current_agent_name}'!")
```

<summary><h3>فاز ۱۳ — Security, Sandboxing & Safety</h3></summary>

**📁 `Security-Sandboxing-Safety/`**

در نرم‌افزار سنتی، خودِ کد تعیین می‌کند چه چیزی مجاز است. در سیستم‌های ایجنتیک، LLM یک **تفسیرکننده‌ی Stochastic** است که می‌تواند دستورات را بر اساس هر ورودی نامعتبری که وارد کانتکستش شود، بازتفسیر کند — و یک ایجنت با محدوده‌ی مشخص را به یک **Confused Deputy** تبدیل کند: او هنوز اعتبارنامه‌های مشروع خودش را دارد، اما فریب می‌خورد که آن‌ها را به نفع مهاجم به‌کار ببرد.

```
              UNTRUSTED INPUT (User Prompt / RAG / Web Result)
                          │
                Agent LLM (Context Injection)
                          │
              [Confused Deputy Vulnerability]
     ┌─────────────────────┼─────────────────────┐
Direct/Indirect      Unauthorized             Dynamic Code
Prompt Injection     Tool Calling             Execution
```

- **تزریق مستقیم (Jailbreak)** — کاربر مستقیماً سعی می‌کند مدل را از گاردریل‌هایش خارج کند
- **تزریق غیرمستقیم (بسیار خطرناک‌تر)** — دستورالعمل‌های مخرب که در محتوای *بازیابی‌شده* جاسازی شده‌اند (یک PDF مسموم در پایپ‌لاین RAG شما، متن یک صفحه‌ی وب اسکرپ‌شده) که ایجنت آن‌ها را مشروع تلقی می‌کند

**دفاع لایه‌به‌لایه (Defense-in-Depth)**، در چهار سطح:

```
Layer 1: Input Sanitization & Pre-execution Policy Check
Layer 2: Privilege Separation (Dual-LLM Pattern)
Layer 3: AST / Static Code Analysis (before runtime execution)
Layer 4: Infrastructure Isolation (gVisor / MicroVM Sandbox)
```

- **الگوی Dual-LLM** — یک *LLM بدون امتیاز* محتوای خام و نامعتبر را پردازش/خلاصه می‌کند (بدون هیچ دسترسی به ابزار)؛ یک *LLM ممتاز* جداگانه، بر اساس خروجی فیلترشده و ساختاریافته‌ی مرحله‌ی قبل، درباره‌ی اجرای ابزارها تصمیم می‌گیرد
- **حداقل‌دسترسی برای ابزارها (Least Privilege)** — ابزارهای دانه‌ریز و محدود (`git_clone`، `read_file`) به‌جای یک `execute_bash` عمومی؛ عملیات مخرب (حذف در دیتابیس، پرداخت، ایمیل انبوه) نیازمند تأیید صریح انسان (Human-in-the-Loop) هستند

**مقایسه‌ی سندباکس‌ها برای اجرای کدهای تولیدشده توسط LLM:**

| سندباکس | سطح ایزوله‌سازی | Latency راه‌اندازی | مقاومت در برابر Kernel Escape |
|---|---|---|---|
| `exec()` / `eval()` | صفر | ۰ میلی‌ثانیه | هیچ — ریسک کامل RCE |
| Docker (استاندارد) | متوسط | ~۱۰۰ تا ۵۰۰ میلی‌ثانیه | ضعیف — اشتراک Kernel با Host |
| **gVisor / Firecracker** | **بسیار بالا** | **~۵ تا ۲۰ میلی‌ثانیه** | **بسیار بالا — Kernel مجازی** |
| WebAssembly (WASM) | بالا | کمتر از ۱ میلی‌ثانیه | بالا — محدود به Sandbox حافظه |

استاندارد صنعت: **gVisor یا Firecracker MicroVMs**، همراه با `--network none` برای ایزوله‌سازی کامل شبکه در هر مسیر اجرای کد نامعتبر.

پیاده‌سازی مرجع، یک **Guard استاتیک بر پایه‌ی AST** است که درخت نحوی کد پایتون تولیدشده توسط LLM را *پیش از* رسیدن به سندباکس پیمایش می‌کند و Importهای ممنوع (`os`، `subprocess`) و فراخوانی‌های ممنوع (`eval`، `exec`، `open`، `__import__`، `compile`) را کاملاً رد می‌کند.

```python
def visit_Call(self, node: ast.Call):
    if isinstance(node.func, ast.Name):
        if node.func.id in {"eval", "exec", "open", "__import__", "compile"}:
            self.violations.append(f"Forbidden function call: '{node.func.id}()'")
```

<summary><h3>فاز ۱۴ — Observability, Tracing & Evaluation (نسخه‌ی دوم)</h3></summary>

**📁 `Observability-Tracing-Evaluation/`**

لایه‌ی نهایی پایش، روی سیستم *کاملاً یکپارچه‌شده* اعمال می‌شود — ایمن، چند-ایجنتی، مجهز به حافظه. متریک‌های سنتی APM (CPU، وضعیت HTTP، Latency) به‌طور ساختاری در برابر شکست‌های خاص ایجنتیک کور هستند: یک درخواست می‌تواند `200 OK` برگرداند در حالی که ایجنت در ۵ ابزار اشتباه حلقه زده، ۵ دلار توکن سوزانده، و یک پاسخ کاملاً هالوسینیت‌شده تولید کرده است.

```
             USER AGENT REQUEST
                     │
        Execution Graph / DAG
     (Spans: Thought → Tool → Memory)
                     │
   ┌─────────────────┼─────────────────┐
Tracing         Cost & Token        Evaluations
(Trajectory)     Attribution        (LLM-Judge)
```

پایش بر سه ستون استوار است: **Trajectory Tracing** (هر گام درخت تصمیم به‌عنوان یک Span ثبت می‌شود)، **Token & Cost Attribution** (تفکیک دقیق هزینه بر اساس مدل/ابزار/کاربر)، و **Behavioral Evals** (کیفیت استدلال + تشخیص Drift).

استاندارد شده بر پایه‌ی **قراردادهای معنایی OpenTelemetry برای GenAI** — هر اجرای ایجنت یک `Trace` است که به `Span`های تایپ‌شده تجزیه می‌شود:

| نوع Span | ویژگی‌های کلیدی |
|---|---|
| Agent / Workflow | `trace_id`، `total_cost`، `total_tokens`، `status` |
| LLM Inference | `model_name`، `prompt_tokens`، `completion_tokens`، `temperature` |
| Tool Execution | `tool_name`، `input_args`، `output_result`، `latency_ms`، `error` |
| Memory Access | `retrieval_query`، `top_k`، `vector_scores`، `latency_ms` |

**ماتریس ارزیابی**، هم به‌صورت Offline (CI/CD) و هم Online (Guardrail پروداکشن):
- **Trajectory Efficiency** — نسبت تعداد گام‌های بهینه به تعداد گام‌های واقعی؛ امتیاز پایین نشان‌دهنده‌ی سردرگمی ایجنت در انتخاب ابزار است
- **Tool Selection Accuracy** — آیا ابزار درست را با ورودی‌های معتبر Schema فراخوانی کرده؟
- **Faithfulness** — آیا پاسخ نهایی کاملاً بر خروجی ابزار/RAG استوار است، یا بخشی از آن ساختگی است؟
- **Goal Completion Rate** — با LLM-as-a-Judge در برابر درخواست اصلی کاربر سنجیده می‌شود

**الگوی Regression در CI/CD:** نگه‌داشتن مجموعه‌ای از حداقل ۵۰ **Golden Test Case** (سناریوی ورودی + مسیر انتظار‌رفته + خروجی مرجع)؛ اجرای خودکار آن‌ها در Staging قبل از Merge؛ **شکست Build** اگر Goal Completion یا Tool Accuracy بیش از ۲٪ افت کند. این همان چیزی است که آسیب خاموش یک تغییر یک‌کلمه‌ای در System Prompt یا ارتقای مدل را به رفتار انتخاب ابزار، شناسایی می‌کند.
